## Ejercicio 1: RAG SysOps Vault (01_rag_sysops_vault)

### 1. Introducción y Contextualización
El primer ejercicio consiste en el desarrollo de un almacén de conocimientos local basado en la metodología RAG (*Retrieval-Augmented Generation*) pura, diseñado específicamente para no depender de modelos generativos de IA en esta fase. Este sistema sirve para indexar apuntes académicos y manuales técnicos de administración de sistemas (SysOps) con el fin de realizar búsquedas semánticas basadas en la similitud matemática de los textos, en lugar de búsquedas tradicionales por palabras clave exactas. El contexto en el que se aplica este desarrollo es el de soporte técnico o documentación interna corporativa, permitiendo a los administradores localizar comandos específicos de despliegue, configuración de servidores o reglas de cortafuegos de manera ágil a partir de lenguaje natural y sin transferir ningún tipo de datos a servidores externos.

### 2. Uso del Programa — Instalación y Arranque
Para hacer uso de este programa, en primer lugar se debe preparar el entorno virtual local mediante comandos en consola que aíslen las dependencias del proyecto. Para ello, accedemos a la carpeta correspondiente, creamos el entorno ejecutando `python3 -m venv .venv` y lo activamos con `source .venv/bin/activate` en macOS. Una vez inicializado este entorno, se procede a instalar las librerías necesarias especificadas en el archivo de requerimientos escribiendo `pip install -r requirements.txt`. Seguidamente, es fundamental verificar que el servidor local de inferencia de Ollama se encuentra en ejecución en segundo plano (`ollama serve`) y que se ha descargado el modelo específico de vectorización semántica mediante la instrucción `ollama pull nomic-embed-text`. Para iniciar el backend del servicio web, se ejecuta la herramienta Uvicorn especificando el puerto recomendado: `uvicorn app:app --reload --port 8101`.

### 3. Uso del Programa — Ingesta e Interacción
Una vez que el servidor backend está en funcionamiento en la dirección `http://localhost:8101`, el usuario puede interactuar con el sistema a través de su interfaz web gráfica optimizada en tonos cálidos ámbar. El flujo habitual de trabajo comienza al hacer clic en el botón de vaciado de índices, seguido de la carga del documento de apuntes en texto plano y la pulsación del botón de ingesta de datos, proceso que vectoriza automáticamente cada fragmento y lo almacena localmente en un archivo estructurado. A continuación, el usuario puede introducir cualquier consulta técnica en el cuadro de búsqueda general y filtrar opcionalmente por una categoría específica como "nginx" o "docker". Al procesar la búsqueda, la interfaz web devuelve una lista de resultados ordenados por puntuación de relevancia (*score*), destacando los términos coincidentes mediante etiquetas HTML `<mark>` que facilitan la lectura rápida.

### 4. Tecnologías Utilizadas — Componentes y Vectores
El software está estructurado sobre el framework FastAPI para la API de backend, apoyado en la librería estándar Uvicorn para servir las peticiones HTTP y en el paquete HTTPX para la comunicación interna con Ollama. El almacenamiento de vectores se realiza de forma directa en disco empleando estructuras JSON, mientras que la vectorización real corre a cargo del modelo local `nomic-embed-text` a través de la API de Ollama, que transforma cada fragmento de texto en un vector numérico de 768 dimensiones. Para calcular la similitud semántica entre la búsqueda introducida y los apuntes almacenados, se implementan fórmulas algebraicas de similitud del coseno usando la biblioteca NumPy, la cual calcula la proyección de la consulta sobre los vectores de almacenamiento dividiendo su producto escalar por el producto de sus normas.

### 5. Tecnologías Utilizadas — Código y Errores Comunes
En cuanto a la arquitectura del código fuente, el archivo `store.py` encapsula la lógica de almacenamiento e indexación donde la función principal `search` recibe la consulta vectorizada, realiza el producto de matrices vectoriales en NumPy y devuelve los índices con mayor coincidencia de similitud. Uno de los errores más comunes al implementar y ejecutar esta aplicación es el fallo en la conexión con Ollama (`ConnectionRefusedError`), que suele ocurrir si el demonio `ollama serve` no ha sido iniciado antes del servidor uvicorn. Otro fallo habitual es la inconsistencia de dimensiones al intentar calcular similitudes debido a que el modelo local de embeddings no es el correcto, problema que se solventa forzando la descarga correcta del modelo `nomic-embed-text` y limpiando el archivo persistente JSON para regenerar los vectores.

### 6. Cierre y Conclusión
En conclusión, el proyecto RAG SysOps Vault ilustra cómo es posible estructurar un sistema de recuperación de información semántica preciso y local utilizando recursos de procesamiento optimizados. Los puntos clave de este ejercicio giran en torno al preprocesamiento de textos, la vectorización mediante embeddings y la búsqueda de proximidad por similitud de cosenos, logrando una tasa de recuperación superior a la del índice de texto clásico. Esta práctica enlaza directamente con los conceptos de integración de servicios y consumo de APIs locales estudiados en la asignatura de Servicios y Procesos, demostrando que no es imprescindible transferir información confidencial a nubes de terceros para obtener resultados inteligentes y altamente contextualizados.

## Ejercicio 2: RAG + Asistente IA (02_rag_ia_asistente)

### 1. Introducción y Contextualización
Este ejercicio avanza sobre la base del recuperador semántico integrando un modelo de lenguaje de gran tamaño (LLM) local para crear un asistente inteligente interactivo con capacidades RAG completas. El concepto general consiste en interconectar el motor de búsqueda vectorial del ejercicio anterior con una interfaz conversacional para que, al plantear una duda sobre administración de sistemas, el backend recupere primero los apuntes relevantes y luego los envíe al LLM como contexto restringido para que redacte una respuesta coherente y justificada. Este enfoque resuelve el problema de las alucinaciones en los modelos de IA y resulta vital en entornos empresariales donde se requiere respuestas veraces basadas exclusivamente en manuales técnicos oficiales, garantizando al mismo tiempo la privacidad absoluta de los datos organizacionales.

### 2. Uso del Programa — Preparación del Entorno
La puesta en marcha de este asistente inteligente sigue un proceso de preparación del entorno local en la consola de comandos de manera análoga al anterior. El usuario debe acceder a la carpeta correspondiente, activar su entorno virtual dedicado y ejecutar la instalación de librerías mediante el comando `pip install -r requirements.txt` para asegurar que las dependencias estén actualizadas. En este caso concreto, además del modelo de vectorización, es imprescindible tener descargado en Ollama el modelo generativo de lenguaje, recomendándose la ejecución de `ollama pull llama3.1:8b` u otro modelo conversacional ligero compatible. Posteriormente, se inicia el servidor de FastAPI mediante el comando `uvicorn app:app --reload --port 8102`, asegurándose de no colisionar con el puerto asignado al ejercicio previo.

### 3. Uso del Programa — Flujo conversacional
Tras abrir el navegador en la URL `http://localhost:8102`, el usuario visualizará una interfaz de chat moderna de color azulado en la que primero deberá indexar el archivo de apuntes técnicos precargado como ejemplo. Una vez indexado el almacén semántico, el alumno introduce una pregunta técnica (por ejemplo: "¿Cómo expongo mi API interna usando Nginx en local?"). El backend procesa la petición en dos fases secuenciales: en primer lugar, consulta la base de datos vectorial local para extraer los fragmentos con mayor puntuación semántica y, en segundo lugar, concatena dichos fragmentos dentro de un prompt estructurado de sistema que es enviado a Ollama. Al cabo de unos segundos, la pantalla muestra la respuesta redactada de forma profesional junto a un panel desplegable de "Fuentes" donde se detallan las citas y las puntuaciones de similitud de los apuntes originales utilizados para fundamentar la respuesta.

### 4. Tecnologías Utilizadas — Integración e Inferencia
El stack tecnológico de esta aplicación combina la potencia del backend en Python estructurado en FastAPI con la flexibilidad de Ollama para la inferencia de inteligencia artificial sin conexión. El sistema realiza la búsqueda semántica en memoria o mediante el gestor vectorial simplificado implementado en `store.py` utilizando embeddings de texto generados por `nomic-embed-text`. Para la parte generativa, el script interactúa con la API REST de Ollama (`/api/generate`) enviando una cadena JSON estructurada con parámetros de temperatura y el contexto delimitado por marcadores de sistema. La interfaz de usuario aprovecha HTML5 semántico y CSS personalizado con transiciones fluidas de carga que indican visualmente que la IA local está generando tokens en tiempo real.

### 5. Tecnologías Utilizadas — Implementación y Depuración
Analizando el código del módulo `app.py`, se observa una ruta POST `/ask` que toma la consulta del cliente, llama a la clase `DocumentStore` para recuperar los contextos más relevantes y ensambla un prompt con instrucciones explícitas de no responder si la información no figura en las fuentes. Un error habitual que surge en este proyecto es la obtención de respuestas vacías o errores de tiempo de espera agotado (`timeout`) en sistemas con hardware limitado si se intenta utilizar modelos excesivamente grandes. Para evitar este contratiempo, se aconseja monitorizar el uso de RAM del sistema, recurrir a modelos cuantizados eficientes como `llama3:8b-instruct-q4_K_M` o ajustar la propiedad `timeout` en las llamadas HTTPX a 60 segundos o más para tolerar la latencia de la CPU en la generación de tokens.

### 6. Cierre y Conclusión
En resumen, la aplicación RAG + Asistente IA ilustra la automatización de la síntesis de información técnica contextualizada mediante inteligencia artificial de consumo local. Los conceptos fundamentales de la práctica abarcan el diseño de plantillas de prompts de sistema (*Prompt Engineering*), el control de latencia en llamadas a APIs locales y la integración de mecanismos de verificación de fuentes. Este módulo se vincula con los contenidos de programación concurrente y comunicaciones por red del plan de estudios de DAM2, mostrando un caso de uso práctico donde se coordinan servicios desacoplados para dotar de inteligencia artificial robusta y privada a herramientas internas de la empresa.

## Ejercicio 3: RAG Empaquetado en Docker (03_rag_empaquetado_docker)

### 1. Introducción y Contextualización
El tercer proyecto consiste en el empaquetado, securización y distribución del servicio web de almacenamiento semántico RAG utilizando la tecnología de contenedores Docker. El concepto general radica en empaquetar la aplicación de Python con todas sus dependencias, variables de entorno y archivos estáticos de forma que sea 100% portable y aislada de las características del sistema operativo del servidor anfitrión. En el contexto de la administración de sistemas moderna y los despliegues en la nube, este ejercicio simula la puesta en producción de microservicios corporativos bajo estándares profesionales de integración y despliegue continuo (CI/CD), securizando las comunicaciones mediante el uso de llaves de API restringidas e implementando persistencia de datos persistente.

### 2. Uso del Programa — Construcción e Inicio
Para utilizar este programa empaquetado, no se requiere la configuración de entornos virtuales locales ni la instalación manual de dependencias de Python, ya que todo el entorno se construye automáticamente dentro del motor de contenedores de Docker. En primer lugar, es necesario abrir Docker Desktop en la máquina host y abrir una terminal en el directorio del proyecto `03_rag_empaquetado_docker`. A continuación, se construye y levanta la pila de contenedores ejecutando el comando de orquestación `docker compose up --build`, el cual lee las instrucciones del archivo `docker-compose.yml` y descarga las imágenes base para compilar el servicio. El backend web se iniciará en el puerto mapeado `8010`, y se puede comprobar su estado de salud accediendo a la ruta `http://localhost:8010/health`.

### 3. Uso del Programa — Clientes y Verificación
La interacción con este servicio empaquetado se realiza habitualmente a través de peticiones HTTP API REST estructuradas utilizando clientes ligeros en consola de comandos o mediante el script automatizado `./demo_client.sh` provisto. Debido a la capa de seguridad implementada en FastAPI, cualquier petición HTTP dirigida a los endpoints `/ingest` o `/search` debe incluir obligatoriamente la cabecera `X-API-Key` con el valor secreto `dam2-secret-2026`; de lo contrario, el cortafuegos de la aplicación interceptará la llamada y devolverá un código de error HTTP `401 Unauthorized`. Para verificar el correcto funcionamiento de los volúmenes dockerizados de datos, el usuario puede poblar la base de datos, apagar el contenedor con `docker compose down`, volver a levantarlo con `docker compose up` y realizar una consulta para corroborar que la información no se ha perdido.

### 4. Tecnologías Utilizadas — Docker y Mapeo
El stack de empaquetado utiliza un archivo de especificaciones `Dockerfile` multilinea que parte de una imagen optimizada de Python (`python:3.10-slim`) para minimizar el peso total del contenedor, instalando las librerías NumPy, FastAPI, Uvicorn y HTTPX. La orquestación corre a cargo de `docker-compose.yml`, que define el servicio web de la aplicación, expone el puerto `8010` mapeado hacia el puerto `8000` interno del contenedor, y declara la variable de entorno de seguridad `API_KEY`. Para lograr la retención de los apuntes vectorizados a lo largo del ciclo de vida del contenedor, el descriptor YAML configura un volumen de tipo directorio local mapeando el directorio de datos `./data` del host hacia el directorio `/app/data` interno del contenedor Docker.

### 5. Tecnologías Utilizadas — Control de Seguridad y Errores
El control de accesos se realiza a nivel de código en `app.py` utilizando el sistema de dependencias de FastAPI (`Security` y `APIKeyHeader`), que evalúa de forma asíncrona la validez de la cabecera enviada contra la variable configurada. Un error muy común en este escenario de despliegue es el fallo de conexión con el host de Ollama (`http://localhost:11434`), ya que "localhost" dentro del contenedor hace referencia al propio contenedor y no a la máquina anfitriona donde corre el servicio de IA. Para solucionar este problema de red en entornos macOS o Windows, se debe configurar la dirección de Ollama en las variables del contenedor como `http://host.docker.internal:11434` o establecer el parámetro `network_mode: "host"` en sistemas basados en Linux.

### 6. Cierre y Conclusión
En conclusión, el empaquetado del servicio RAG en Docker representa un paso decisivo hacia la profesionalización y estandarización del software en el ámbito de SysOps. Los pilares de este ejercicio demuestran la importancia de la portabilidad del código, la persistencia de datos mediante volúmenes compartidos y la securización de interfaces programáticas mediante tokens de API. Estos conocimientos forman parte esencial de la gestión de contenedores y despliegues en servidores Linux del programa académico, enseñando al alumno a diseñar microservicios desacoplados, autocontenidos y listos para ser distribuidos en clústeres de orquestación de producción empresarial.

## Ejercicio 4: CSS3D Stack Explorer (04_css3d_stack_explorer)

### 1. Introducción y Contextualización
El cuarto ejercicio abandona el backend de IA para centrarse en el diseño y renderizado tridimensional avanzado en el lado del cliente utilizando exclusivamente capacidades modernas de CSS3D. El concepto general del proyecto es la creación de un salpicadero de monitorización interactivo (dashboard) estructurado en torno al stack de tecnologías de SysOps estudiadas en el curso, renderizando modelos espaciales en 3D acelerados por hardware. Este explorador sirve para visualizar de forma atractiva, didáctica e inmersiva la interconexión entre componentes de software (como nginx, Docker, cortafuegos e inteligencia artificial), utilizándose en contextos de desarrollo frontend de alta gama donde la experiencia de usuario y el impacto visual son prioritarios para la presentación de proyectos o productos tecnológicos.

### 2. Uso del Programa — Acceso y Navegación
Para ejecutar este programa de frontend no se requiere la presencia de servidores de bases de datos ni backends complejos, puesto que es una aplicación estática autoliquidable basada en la web. El usuario puede abrir la herramienta abriendo directamente el archivo `index.html` en cualquier navegador web moderno o iniciando un servidor web estático básico en local mediante la instrucción en consola `python3 -m http.server 8085`. Una vez cargado el panel en la pantalla, la navegación se inicia automáticamente mediante la visualización de un cubo tridimensional translúcido que rota en bucle sobre sí mismo mostrando diferentes iconos representativos de cada tecnología en sus seis caras.

### 3. Uso del Programa — Ajustes y Controles
El usuario puede interactuar exhaustivamente con el entorno 3D manipulando la barra lateral izquierda, la cual incluye controles para cambiar entre tres escenas: el "Cubo Stack", el "Carrusel 3D" de tarjetas, o la "Pila de Capas" de arquitectura. A través de barras deslizantes, es posible ajustar a mano los ángulos de rotación de los ejes X e Y, cambiar el punto focal de perspectiva en píxeles y regular la velocidad del giro automático de la escena. Asimismo, al hacer clic y arrastrar con el puntero del ratón o el dedo sobre la escena tridimensional, el usuario puede manipular manualmente la orientación con un efecto de inercia muy realista, activando al mismo tiempo en el terminal Unix inferior un volcado de telemetría y comandos de auditoría en tiempo real para la tecnología inspeccionada.

### 4. Tecnologías Utilizadas — Transformaciones y 3D
La arquitectura gráfica de esta aplicación está fundamentada en el uso intensivo de transformaciones 3D de hojas de estilo en cascada (CSS3), aprovechando propiedades críticas como `transform-style: preserve-3d` para heredar las coordenadas espaciales en contenedores hijos. Las caras del cubo y los elementos del carrusel se distribuyen y espacian de forma exacta utilizando la propiedad `transform` combinada con funciones de traslación tridimensional `translateZ` y rotación trigonométrica `rotateY/rotateX`. Las sombras dinámicas en el suelo se logran posicionando un elemento plano difuminado mediante filtros de desenfoque (`filter: blur`) que se escala matemáticamente de acuerdo con la inclinación del cubo utilizando variables CSS actualizadas en tiempo real desde JavaScript.

### 5. Tecnologías Utilizadas — Lógica JS y Accesibilidad
La interactividad fluida de la escena es coordinada mediante lógica en JavaScript en `script.js`, el cual captura eventos del ratón (`mousedown`, `mousemove`, `mouseup`) para calcular la aceleración de arrastre del usuario e implementa un bucle de animación optimizado mediante `requestAnimationFrame` que evita el consumo excesivo de CPU. Un error común al desarrollar efectos 3D en navegadores es el fallo de "recorte de caras" (*flickering* o desaparición de caras) durante la rotación, el cual se evita definiendo correctamente la propiedad `backface-visibility: hidden` en las clases de las caras y configurando el valor de perspectiva a nivel de contenedor principal. La accesibilidad está resuelta mediante controles y estilos alternativos que se disparan al detectar la media query de accesibilidad `prefers-reduced-motion` o el activador manual correspondiente.

### 6. Cierre y Conclusión
En conclusión, el proyecto CSS3D Stack Explorer demuestra el potencial de las tecnologías web estándar para la creación de interfaces de usuario ricas, dinámicas y tridimensionales con alto rendimiento visual. Los aspectos fundamentales tratados en la práctica comprenden el posicionamiento espacial en 3D, la captura interactiva de gestos con inercia física y el diseño estructurado bajo criterios modernos de accesibilidad universal. Esta entrega cierra el ciclo académico del examen enlazando los conocimientos del lado cliente con las arquitecturas backend previamente estudiadas, ofreciendo un panel de visualización premium y unificado que pone en valor la importancia de una presentación estética e interactiva sobresaliente en la ingeniería de software actual.
